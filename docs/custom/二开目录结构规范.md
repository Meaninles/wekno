# 二开目录结构规范

本文档用于约束 WeKnora 本地二次开发的目录组织。目标是让二开代码和原生代码清晰分离，减少后续升级 WeKnora 时的冲突。

## 总原则

- 二开能力优先放在 `custom/` 或 `internal/custom/`，不要把大段业务逻辑散落到原生目录。
- 能用独立旁路服务解决的，优先放到 `custom/services/<module>/`。
- 必须嵌入 WeKnora 后端时，放到 `internal/custom/modules/<module>/`。
- 必须嵌入 WeKnora 前端时，放到 `frontend/src/custom/modules/<module>/`。
- 原生代码只保留少量稳定注册点，例如路由注册、依赖注册、生命周期启动。
- 每个二开模块独立成目录，模块之间通过清晰接口交互，不互相穿透实现。

## 原生代码修改边界

后端尽量只在这些位置增加统一入口：

```text
internal/container/container.go
internal/router/router.go
```

入口形式应类似：

```go
custom.RegisterProviders(container)
custom.RegisterRoutes(v1, guards)
custom.RegisterLifecycle(container)
```

以后新增二开模块时，只改 `internal/custom/bootstrap/` 和对应模块目录，避免继续修改原生路由、服务和业务逻辑。

前端如果需要嵌入菜单或页面，尽量只在这些位置增加统一入口：

```text
frontend/src/router/index.ts
frontend/src/stores/menu.ts
```

新增页面、接口、菜单配置应放在 `frontend/src/custom/modules/<module>/`。

## 命名规范

- API 前缀：`/api/v1/custom/<module>/*`
- 表名前缀：`custom_<module>_*`
- 环境变量前缀：`CUSTOM_<MODULE>_*`
- Docker 服务名：`weknora-custom-<module>`
- 二开 migration 编号建议从 `900000` 以后开始。
