# 统一身份认证与默认配置二开实现说明

## 目录边界

- 后端二开逻辑位于 `internal/custom/`。
- 前端二开页面位于 `frontend/src/custom/`。
- 自定义迁移参考文件位于 `migrations/custom/`。
- 原生代码只保留必要注册点：
  - `internal/container/container.go` 注册 custom bootstrap。
  - `internal/router/router.go` 挂载 `/api/v1/custom` 路由。
  - `internal/handler/auth.go` 在 `/auth/me` 调用可选 provisioning hook。
  - `internal/middleware/auth.go` 放行 SSO public callback。
  - `frontend/src/views/settings/Settings.vue` 挂载两个设置页入口。
  - `frontend/src/views/auth/Login.vue` 增加可选统一身份认证登录按钮。
  - `frontend/src/views/chat/index.vue` 和 `frontend/src/components/menu.vue` 接入会话已读状态。
  - `frontend/src/router/index.ts` 注册 `/share/chat/:token` 登录态路由，`frontend/src/components/menu.vue` 和移动端聊天页挂载对话分享入口。

## 功能模块

### IAM 登录与组织人员同步

- 后端模块：`internal/custom/modules/iam`
- 管理接口：`/api/v1/custom/iam/*`，仅 SystemAdmin 可访问。
- 登录接口：`/api/v1/custom/iam/sso/*`，浏览器回调前无需 WeKnora token。
- 默认同步计划：每日 `03:10`。
- 同步结果写入二开表 `custom_iam_*`，并按需创建或启停 WeKnora 本地用户。
- SSO 登录成功签发本地登录态后，会同步更新原生用户表 `users.last_login_at`。

#### 生产入口与反向代理要求

- IAM 工作台应用入口配置为：`https://knora.moutai.com.cn/api/v1/custom/iam/sso/entry`
- IAM OAuth 回调地址配置为：`https://knora.moutai.com.cn/api/v1/custom/iam/sso/callback`
- 生产环境不需要对外暴露后端 `8080` 端口；`/api/` 请求应从前端域名进入，再由前端 Nginx 或网关代理到后端。
- 如果前端容器前面还有云网关、负载均衡或外层 Nginx，必须向前端层透传下面的请求头：

```nginx
proxy_set_header Host $host;
proxy_set_header X-Forwarded-Host $host;
proxy_set_header X-Forwarded-Proto $scheme;
```

原因：SSO `entry` 会根据外部访问域名和协议动态生成发给 IAM 的 `redirect_uri`。`X-Forwarded-Host` 用于还原公网域名，`X-Forwarded-Proto` 用于还原公网协议；如果 HTTPS 在外层终止但没有传 `X-Forwarded-Proto: https`，后端可能生成 `http://.../callback`，导致 IAM 回调地址不匹配或浏览器被错误跳回 HTTP。

#### 企业微信电脑端/移动端同域名部署要求

企业微信工作台同一个应用如果同时配置电脑端和移动端入口，生产环境采用同域名不同路径部署：

- 电脑端入口：`https://knora.moutai.com.cn/api/v1/custom/iam/sso/entry`
- 移动端入口：`https://knora.moutai.com.cn/api/v1/custom/iam/sso/entry?client=mobile`
- IAM OAuth 回调地址仍只配置一个：`https://knora.moutai.com.cn/api/v1/custom/iam/sso/callback`

后端 `entry` 会按请求来源生成 IAM `redirect_uri`，并在 `state` 中保存前端回跳地址：

- 未带 `client=mobile`：回跳 `https://knora.moutai.com.cn/`
- 带 `client=mobile`：回跳 `https://knora.moutai.com.cn/mobile/`

因此在 ELB -> Ingress -> Pod 的生产链路中，Ingress 必须在同一个 Host 下按路径分流：

```text
https://knora.moutai.com.cn/api/      -> WeKnora 后端 app service
https://knora.moutai.com.cn/mobile/   -> 移动端 mobile-web service
https://knora.moutai.com.cn/share/    -> 电脑端 web service，或已内置桌面 SPA 的 mobile-web service
https://knora.moutai.com.cn/assets/   -> 桌面 SPA 静态资源
https://knora.moutai.com.cn/          -> 电脑端 web service
```

路径优先级必须保证 `/api/`、`/mobile/`、`/share/` 和 `/assets/` 优先于 `/`，否则 SSO callback、移动端页面、分享页刷新或分享页静态资源可能被错误服务接走。

对话分享链接固定使用同一个路径：

```text
https://knora.moutai.com.cn/share/chat/:token
```

该链接不区分电脑端和移动端。浏览器打开后由同一个分享页根据视口宽度渲染桌面或移动布局。分享页要求 WeKnora 登录态；未登录时会跳转登录页，并通过 `returnTo=/share/chat/:token` 在登录或 SSO 完成后回到原链接。

创建分享接口返回的 `url` 会优先使用 app 侧 `FRONTEND_BASE_URL` 生成公网绝对地址；未配置时返回同域相对路径，由前端按当前域名补全。

移动端 pod 不能只按独立根路径 `/` 站点理解，还必须支持被挂载到 `/mobile/` 前缀下。部署时必须满足以下任一方式：

- Ingress 将 `/mobile/*` rewrite 到 mobile-web service 的 `/`，同时移动端前端构建、资源引用、Nginx fallback 与 rewrite 规则保持一致。
- 或 mobile-web service 自身直接支持 `/mobile/` 前缀，包含 `/mobile/` 的 SPA fallback 和 `/mobile/assets/*` 静态资源访问。

移动端静态资源路径尤其要检查。若 mobile 前端打包后仍请求 `/assets/...`，浏览器会在同域名根路径访问 `https://knora.moutai.com.cn/assets/...`，该请求可能被 Ingress 转给电脑端 web service，导致移动端白屏或资源 404。采用 `/mobile/` 路径部署时，移动端构建应使用 `/mobile/` 作为资源 base，或在 Ingress/Nginx 层做等价的资源路径兼容。

当前项目采用第二种方式：mobile-web service 自身直接支持 `/mobile/` 前缀，不依赖 Ingress rewrite。相关配置必须保持一致：

- `frontend/vite.mobile.config.ts` 使用 `base: "/mobile/"`，打包后的 JS/CSS 和静态资源请求路径为 `/mobile/assets/...`。
- `frontend/src/custom/modules/mobile/router.ts` 使用 `createWebHistory(import.meta.env.BASE_URL)`，移动端 SPA 路由实际落在 `/mobile/chat`、`/mobile/settings`。
- `frontend/src/utils/api-base.ts` 不能把移动端的 `/mobile/` 资源 base 当作 API base；移动端接口仍必须请求根路径 `/api/v1/...`，例如 `/api/v1/auth/me`，不能变成 `/mobile/api/v1/auth/me`。
- `frontend/nginx.mobile.conf` 服务 `/mobile/`、`/mobile/assets/`，移动端 SPA fallback 指向 `/mobile/mobile.html`；同时服务 `/share/` 和 `/assets/`，分享页 fallback 指向 `/index.html`。
- `frontend/Dockerfile.mobile` 会同时执行 `npm run build:mobile` 和 `npm run build`，将 `dist-mobile` 放入 `/usr/share/nginx/html/mobile`，将桌面端 `dist` 放入 `/usr/share/nginx/html`，入口脚本会生成 `/config.js` 和 `/mobile/config.js`。
- `frontend/.dockerignore` 必须保留 `index.html`、`embed.html`、`vite.config.ts` 等桌面端构建入口，否则 mobile-web 镜像无法构建并承载 `/share/`。
- Helm 部署时设置 `mobileWeb.enabled=true`，Ingress 会自动增加 `/mobile -> mobile-web` 路由，`/share/` 默认由电脑端 frontend 的 `/` catch-all 承载；若自定义 Ingress 把 `/share/` 分流给 mobile-web，需要同时把 `/assets/` 分流给 mobile-web，并保证两者优先级高于 `/`。
- 若不用 Helm，可参考 `custom/k8s/mobile-web.yaml`，该清单同样按同域 `/mobile` 前缀部署。分享页默认仍可由主 frontend Ingress 的 `/` catch-all 承载；只有在决定让 mobile-web 承载 `/share/` 时，才需要额外增加 `/share` 和 `/assets` 路由。

### 默认配置中心

- 后端模块：`internal/custom/modules/configcenter`
- 管理接口：`/api/v1/custom/config-center/*`，仅 SystemAdmin 可访问。
- 管理员从源工作区选择模型、向量数据库、解析、存储、网络搜索、MCP 服务。
- 配置下发方式是复制到用户自己的工作区，尽量复用原生 WeKnora 的资源选择和运行逻辑。
- 支持模板变量：`${tenant_id}`、`${user_id}`、`${username}`。

### 会话已读状态

- 后端模块：`internal/custom/modules/sessionstate`
- 前端接口封装：`frontend/src/custom/modules/sessionState/api.ts`
- 接口前缀：`/api/v1/custom/session-state/*`。
- 状态表：`custom_session_read_states`，记录 `tenant_id`、`actor_key`、`session_id` 和 `read_at`。
- `actor_key` 优先使用 Principal storage ID；没有 Principal 时使用真实用户 ID；合成用户或匿名场景使用独立前缀，避免不同入口的已读状态互相串扰。
- `POST /status` 按会话 ID 批量返回最后一条助手消息、生成中状态和未读状态。单次请求最多处理 120 个去重后的 ID。
- `POST /sessions/:session_id/read` 将当前访问者对该会话标记为已读。
- Web 侧边栏和移动端会话抽屉都会在拉取会话列表后刷新状态；打开当前会话、发送消息、恢复或完成流式回复时会标记为已读。

该模块只保存展示层所需的已读水位，不改变原生 `sessions`、`messages` 的归属、删除、置顶和历史加载逻辑。未读判断只在助手消息已完成且已存在读取水位时生效，避免把首次可见的历史会话全部标成未读。

## 运行注意

- 开发环境启动时通过 custom bootstrap 执行 GORM AutoMigrate，避免改动原生 migration runner。
- 生产发布如需显式 SQL 迁移，可参考 `migrations/custom/` 下的 SQL 文件。
- 若多个用户共享同一个 WeKnora tenant，原生资源可见性仍是 tenant 级；本实现默认按用户 home tenant 下发配置。
- 移动端与电脑端同域部署时，IAM 入口必须保留 `client=mobile` 参数；否则 SSO 完成后会回跳电脑端根路径。
