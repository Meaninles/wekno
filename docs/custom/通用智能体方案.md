# WeKnora 通用智能体完整方案（原生配置一致 + Claude Agent SDK 内核）

> 本文档是本轮「通用智能体」实现、检查、修复和调优的唯一基准。  
> 若上下文压缩导致需求不完整，必须先重读本文档再继续。用户要求：每经过三次上下文压缩，主动重读本文档，重新确认目标和约束。

创建日期：2026-06-28  
状态：完整实现、端到端验证和对抗式效果调优已完成。旧 GLM / 旧通用智能体实现均视为开发中间产物，可按需替换或删除，不要求兼容。

## 0. 一句话目标

新增一类 `general-agent` 通用智能体：

- 配置体验与 WeKnora 原生「智能推理」智能体一致。
- 原生智能体已有的知识库检索、检索策略、网络搜索、多模态、工具配置、MCP 服务、Skills 等配置项，在通用智能体中也必须真实生效。
- 额外支持当前项目已有的数据库数据源配置，即 `db_data_sources`，用于 `db_catalog` / `db_schema` / `db_query` 等数据库分析工具。
- 不新增另一类内容型数据源选择。飞书、Notion、语雀等同步型内容数据源仍通过知识库进入检索范围。
- Claude Agent SDK / Python sidecar 负责 agentic loop；WeKnora Go 后端负责配置解析、权限、工具执行、检索、MCP、Skills、数据源、安全和持久化。
- 不影响原有 WeKnora 功能、原生智能体、知识库问答、智能推理、数据源、MCP、Skills 等既有行为。

## 1. 硬性约束

1. 遵循 `docs/custom/二开目录结构规范.md`。
   - 大段二开逻辑放在 `custom/`、`internal/custom/`、`frontend/src/custom/`。
   - 原生代码只保留必要注册点、类型字段、hook 或兼容性最小变更。

2. 旧实现不保留兼容。
   - 可以应改尽改。
   - 可以替换旧 GLM / 旧方案写法。
   - 不需要兼容旧通用智能体内部实现。
   - 但不能破坏原生 WeKnora 功能。

3. 配置必须生效。
   - 凡是创建智能体 UI 中对通用智能体可见的配置项，运行时必须有对应效果。
   - 如果发现原生智能推理链路中某个配置项本身没有完整透传，应补成共享运行时能力，让原生智能推理和通用智能体都受益。

4. Python sidecar 不直接连接 WeKnora 数据层或外部 MCP。
   - Python 只接收工具 schema。
   - Python 工具调用统一回调 Go。
   - Go 侧复用现有权限、租户、密钥、OAuth、审批、检索、MCP、Skills、数据源和审计能力。

5. 完成代码修改后必须重启受影响容器。
   - 先关闭此前仍在运行的本应用实例，再启动更新后的实例。
   - 当前为开发环境，不要求兼容旧实现、旧配置或旧数据库数据。

## 2. 最终用户效果

创建或编辑通用智能体时，左侧配置项与原生「智能推理」智能体保持一致：

- 基本信息
- 提示词
- 模型配置
- 知识库
- 检索策略
- 网络搜索
- 多模态
- 工具配置
- MCP 服务
- 技能 Skills
- 数据源（新增，仅数据库数据源）
- 编辑态共享、IM、嵌入等管理能力沿用原生逻辑

其中「数据源」只表示当前项目已有的数据库分析数据源：

- MySQL
- PostgreSQL
- 现有 `internal/custom/modules/dbanalytics` 管理的数据源

不做独立的「内容型数据源检索选择」。内容型来源通过知识库同步后，由知识库选择和检索策略控制。

## 3. 总体架构

```text
AgentEditorModal / Chat UI
        │
        │ 保存同一份 CustomAgentConfig
        ▼
Go session/qa 短路：agent_type == general-agent
        │
        ▼
internal/custom/modules/generalagent
        │
        ├─ 构建完整运行时配置 RuntimeConfig
        ├─ 复用原生 SearchTargets / ToolRegistry / MCP / Skills / WebSearch / DBAnalytics
        ├─ 导出工具 schema 给 Python
        ├─ 接收 Python 工具回调并在 Go 侧执行
        ├─ 转发 NDJSON 流式事件到 WeKnora StreamManager
        └─ 持久化消息、AgentSteps、产物
        │
        ▼
custom/services/general-agent Python sidecar
        │
        ├─ Claude Agent SDK 作为唯一 agentic loop
        ├─ 动态注册 Go 下发的工具
        ├─ 私有受限文本产物工具
        └─ 工具调用统一回调 Go
```

核心原则：

- `CustomAgentConfig` 是唯一配置源。
- Go 是能力执行和安全边界。
- Python 是推理循环和工具调度层。
- 前端不做两套配置模型。

## 4. 配置一致性要求

### 4.1 基础与模型

| 字段 | 运行时效果 |
|---|---|
| `agent_mode` | 通用智能体必须是 `smart-reasoning` |
| `agent_type` | `general-agent` 时进入 Python sidecar 短路 |
| `system_prompt` | 作为 Claude Agent SDK 主系统提示词的一部分 |
| `system_prompt_id` | 若前端按预设解析为 `system_prompt`，运行时使用解析后的内容 |
| `model_id` | 选择 WeKnora 对话模型，通用智能体复用该模型的 API key / 凭据；Go 侧为 Claude Agent SDK 派生 Anthropic/Claude 风格专用接口 |
| `temperature` | 传给 sidecar runtime summary；当前 `claude-agent-sdk==0.2.110` 无直接 temperature 参数，因此写入运行提示约束，不传未知 CLI flag |
| `thinking` | 传给 sidecar runtime summary，并映射为 Claude Agent SDK `thinking` 参数：true=adaptive/omitted，false=disabled，空值使用 SDK 默认 |
| `max_iterations` | 映射为 Claude SDK 最大轮次 / max_turns |
| `llm_call_timeout` | 控制 Go→Python 和 Python→LLM 调用超时 |
| `multi_turn_enabled` / `history_turns` | 控制历史上下文加载 |

### 4.2 知识库与检索

| 字段 | 运行时效果 |
|---|---|
| `kb_selection_mode` | 控制知识库范围：all / selected / none |
| `knowledge_bases` | selected 模式下的知识库范围 |
| `retrieve_kb_only_when_mentioned` | 无 @ 提及时不暴露或不执行知识库检索 |
| `supported_file_types` | 限制可检索或可读取附件类型 |
| `embedding_top_k` | 控制向量召回数量 |
| `keyword_threshold` | 控制关键词召回阈值 |
| `vector_threshold` | 控制向量召回阈值 |
| `rerank_model_id` | 使用指定 rerank 模型 |
| `rerank_top_k` | 控制 rerank 后结果数量 |
| `rerank_threshold` | 控制 rerank 过滤阈值 |
| `faq_priority_enabled` | FAQ 优先策略 |
| `faq_direct_answer_threshold` | FAQ 直接回答阈值 |
| `faq_score_boost` | FAQ 分数提升 |

实现要求：

- 原生 AgentEngine 和通用智能体共享同一套检索配置解释逻辑。
- 如果现有 Go Agent 工具只读全局 conversation 配置，而没有读 agent 配置，应改成运行时可注入配置。
- 不能出现前端检索策略配置了但通用智能体不生效的情况。

### 4.3 网络搜索

| 字段 | 运行时效果 |
|---|---|
| `web_search_enabled` | 控制是否暴露 `web_search` 工具 |
| `web_search_provider_id` | 指定 provider，空则使用租户默认 provider |
| `web_search_max_results` | 控制搜索结果数 |
| `web_fetch_enabled` | 控制是否暴露或建议使用 `web_fetch` |
| `web_fetch_top_n` | 控制自动抓取数量；Agent 模式下也限制单次 `web_fetch` 最大 URL 数 |

工具执行必须复用 Go 侧现有 WebSearch provider 和密钥管理，Python 不保存搜索 provider API key。

### 4.4 工具配置

`allowed_tools` 是通用智能体可用工具的主开关。

必须覆盖原生工具：

- `thinking`
- `todo_write`
- `grep_chunks`
- `knowledge_search`
- `list_knowledge_chunks`
- `query_knowledge_graph`
- `get_document_info`
- `database_query`
- `data_analysis`
- `data_schema`
- `web_search`
- `web_fetch`
- Wiki 读取与编辑工具
- `db_catalog`
- `db_schema`
- `db_query`
- Skills 相关工具
- MCP 动态工具
- 通用智能体私有产物工具

工具注册规则：

- UI 里未勾选的工具，Python 不应看到。
- 前置条件不满足的工具不能暴露，例如无知识库时不暴露 RAG/Wiki 工具。
- Web 搜索工具由 `web_search_enabled` 控制。
- MCP 工具由 MCP 配置控制。
- Skills 工具由 Skills 配置和沙箱状态控制。
- DB 数据源工具由 `db_data_sources` 和 `allowed_tools` 共同控制。

### 4.5 MCP 服务

| 字段 | 运行时效果 |
|---|---|
| `mcp_selection_mode` | all / selected / none |
| `mcp_services` | selected 模式下的 MCP 服务 |

实现要求：

- 复用 Go 侧现有 MCP 能力。
- 复用已有密钥、OAuth、stdio/http/sse transport、审批、超时、重连、工具名规范、防注入前缀、图片结果限制。
- Python 不直连 MCP。
- MCP 工具以动态工具形式暴露给 Claude。
- 审批型 MCP 工具必须走现有审批机制，不能绕过。

实现测试应至少配置并验证几个常见 MCP：

- Time MCP
- Filesystem MCP（安全目录）
- Fetch MCP
- Memory / KV MCP
- SQLite MCP

测试服务应通过 WeKnora 原生 MCP 配置登记；通用智能体实现不依赖任何测试 MCP 的硬编码。

### 4.6 Skills

| 字段 | 运行时效果 |
|---|---|
| `skills_selection_mode` | all / selected / none |
| `selected_skills` | selected 模式下允许的 Skills |

实现要求：

- 复用现有 Skills 元数据和沙箱机制。
- 支持 Progressive Disclosure：先把 Skills metadata 提供给模型，需要时读取具体 Skill 内容。
- `read_skill` 和 `execute_skill_script` 是否暴露，取决于 Skills 是否启用、沙箱是否启用、allowed_tools 是否允许。
- 通用智能体提示词必须明确：Skills 是可按需调用的专业工作流，不是强制全部读入上下文。

### 4.7 多模态与附件

| 字段 | 运行时效果 |
|---|---|
| `image_upload_enabled` | 控制图片输入是否允许进入通用智能体 |
| `vlm_model_id` | 非视觉对话模型时用于图片描述；MCP 图片结果也可用其转文本 |
| `image_storage_provider` | 复用现有上传存储选择 |
| `audio_upload_enabled` | 控制音频输入 |
| `asr_model_id` | 控制音频转写模型 |
| `supported_file_types` | 控制附件读取和知识文件选择 |

实现要求：

- 原生输入处理链路已生成 `ImageURLs`、`ImageDescription`、`Attachments` 时，通用智能体应正确消费。
- 支持视觉的模型可直接接收图片；不支持时使用描述文本。
- 附件读取通过 Go 回调，不让 Python 自行访问存储。

### 4.8 数据库数据源

仅支持当前项目已有数据库数据源：

| 字段 | 运行时效果 |
|---|---|
| `db_data_sources` | 限制 `db_catalog` / `db_schema` / `db_query` 可访问的数据源 |

实现要求：

- 前端通用智能体显示「数据源」页，选择 MySQL/PostgreSQL 数据源。
- 不新增其他内容型数据源字段。
- 不做飞书、Notion 等内容型数据源的单独选择。
- DB 工具执行复用 `internal/custom/modules/dbanalytics`。
- 查询工具必须保持只读和安全限制。
- 共享智能体运行时使用智能体归属租户的数据源范围，不泄露调用者私有数据源。

### 4.9 产物

字段：

- `enable_artifacts`

当前实现：

- 关闭 `enable_artifacts` 时，不暴露通用智能体私有产物工具。
- 开启后，Python sidecar 暴露 `create_artifact`，用于生成通用可下载产物。
- 不设置产物格式白名单，不提供用户可配置的最多文件数；通用智能体按用户请求和当前能力自由生成产物，但模型侧必须遵守产品返回预算。
- `docx`、`xlsx`、`pdf` 会渲染为真实办公/文档格式；用户请求旧版 `doc` / `xls` 时，最终输出现代 `docx` / `xlsx`。其他扩展名按 UTF-8 文本内容落盘。
- Claude SDK 提示词必须说明产物返回策略：最多创建/注册 5 个产物，注册产物合计必须小于 128MB；不允许模型故意多生成后依赖后端丢弃。
- Claude SDK 提示词必须要求模型在创建多个产物前按用户价值/任务重要性排序，并按“最重要 → 最不重要”的顺序调用 `create_artifact`，避免低价值附件排在主产物前面。
- 返回给前端前，sidecar 必须执行脚本级最终校验：按 `create_artifact` 调用顺序保留最多 5 个且累计小于 128MB 的完整文件，丢弃后续文件，并在前端下载/入库产物卡片显示提醒。这里按整文件丢弃，不做文件内字节截断，避免破坏 docx/xlsx/pdf 等二进制格式。
- 不做自动 ZIP 打包。只有当用户明确要求 ZIP 作为产物本身时，Claude 才能把 ZIP 当作最多 5 个文件之一生成，且总大小仍必须小于 128MB。
- sidecar 只写入 run 私有目录，Go 再下载、校验、持久化并提供下载 URL。
- 前端仅展示「允许生成文件」开关，不展示格式/数量限制配置。

通用智能体的产物能力不是文档处理智能体替代品：它负责“新建通用产物文件”。如果要对用户上传的 Word/Excel/PDF 做结构继承、版式保持、复杂修改、旧格式输入转换和确定性校验，应继续走文档处理智能体或后续新增独立文档工具。

后续如需要真正的代码解释器或复杂文件加工，应新增独立沙箱工具，而不是让 sidecar 直接执行任意 Python：

- `run_python`：必须在受控容器/沙箱中执行，限定 CPU、内存、运行时长、网络、输出目录。
- `collect_files`：只允许收集沙箱输出目录内文件。
- 二进制产物（xlsx/docx/pdf/png）必须走 MIME、扩展名、大小和恶意文件校验。

## 5. Go 侧实现方案

### 5.1 通用运行时适配层

在 `internal/custom/modules/generalagent` 中新增：

```text
module.go              # 模块说明
types.go               # payload、event、artifact、tool request 类型
service.go             # RunAsCore、payload 构建、stream 转发、持久化
client.go              # Go → Python HTTP/NDJSON client
handler.go             # artifact 与 internal tool callback endpoints
tool_bridge.go         # 工具导出、工具调用、工具结果转码
```

同时在原生 service 层暴露共享能力：

- `SessionService.BuildAgentRuntimeConfig(ctx, req)`：复用原生 `buildAgentConfig`，不执行 Go AgentEngine。
- `AgentService.CreateToolRegistry(ctx, config, chatModel, rerankModel, sessionID)`：复用原生工具注册逻辑，不创建 Go AgentEngine。

### 5.2 运行时构建

`RunAsCore` 必须：

1. 校验 `req.CustomAgent.Config.AgentType == general-agent`。
2. 解析智能体归属租户，尤其是共享智能体场景。
3. 调用 `EnsureDefaults`。
4. 解析模型。
   - 通用智能体不禁止用户选择 GLM/Zhipu/OpenAI-compatible 等普通对话模型。
   - 运行时复用选中模型的 API key / 凭据，优先读取 `remote_model_name` 作为真实模型名。
   - 已知 provider 自动派生 Anthropic/Claude 风格接口：Anthropic 直连、DeepSeek 使用 `/anthropic` 专用接口、Zhipu/GLM 使用 BigModel 的 Claude/Anthropic 风格接口。
   - 未知 provider 可通过模型 `extra_config.general_agent_claude_base_url` / `general_agent_claude_model_name` 或 `CUSTOM_GENERAL_AGENT_CLAUDE_*` 环境变量指定专用接口；这不是 UI 禁选，而是运行时端点配置。
5. 解析知识库和 `SearchTargets`。
6. 应用检索策略字段。
7. 解析 WebSearch provider。
8. 解析 DB 数据源范围。
9. 解析 MCP 服务和 MCP 工具。
10. 解析 Skills。
11. 注册所有有效工具到本次 run 的 tool registry。
12. 生成 `runtime_tools` 下发给 Python。
13. 传递完整运行时配置给 Python。

### 5.3 工具调用接口

新增内部接口：

```text
POST /api/v1/custom/general-agent/internal/tools/call
```

请求字段：

```json
{
  "tenant_id": 1,
  "source_tenant_id": 1,
  "user_id": "u",
  "session_id": "s",
  "assistant_message_id": "m",
  "request_id": "req",
  "run_id": "r",
  "tool_call_id": "tc",
  "tool_name": "knowledge_search",
  "arguments": {}
}
```

响应字段：

```json
{
  "ok": true,
  "success": true,
  "output": "...",
  "data": {},
  "images": [],
  "error": ""
}
```

要求：

- 工具调用必须基于本次 run 已注册的工具，不允许 Python 任意传工具名调用未授权能力。
- 每个 run 应保存工具 registry 或可重建工具 registry。
- 工具输出要截断和清洗，避免过长或注入。
- MCP 输出、Web 输出、外部数据输出统一标记为不可信数据。

## 6. Python sidecar 实现方案

位置：`custom/services/general-agent/`

### 6.1 Schema

`ChatPayload` 增加：

```python
run_id: str
runtime_tools: list[RuntimeToolSpec]
runtime_config: RuntimeConfig
```

`RuntimeToolSpec`：

```python
name: str
description: str
input_schema: dict
source: str  # builtin / retrieval / web / mcp / skill / db / private
requires_approval: bool
```

### 6.2 动态工具注册

- Go 下发什么工具，Python 注册什么工具。
- 每个动态工具执行时统一调用 `CallbackClient.call_tool(name, args)`。
- Python 不实现 WeKnora 检索、WebSearch、MCP、Skills、DB 数据源业务逻辑。

当前保留 Python 私有工具：

- `create_artifact`

该工具必须由 `enable_artifacts` 控制。`run_python` / `collect_files` 等代码解释器类工具不在当前阶段启用，避免未沙箱化执行。

### 6.3 提示词

通用智能体传给 Claude SDK 的上下文由四部分组成：

1. 用户配置的 `system_prompt`。
2. 当前用户在 WeKnora 对话框输入的原文 `user_request`。
3. 用户在 WeKnora 前端可见/可理解的上下文 `visible_context`，包括智能体配置、模型、知识库/文件、数据源、MCP、Skills、附件、图片、引用、历史消息等。
4. 当前启用能力摘要与工具目录。

必须包含：

- 当前任务的最重要目标是用户对话输入的原文；该原文必须作为 `user_request` 放在 Claude SDK 用户消息顶部，并标记为最高优先级。其他上下文只能辅助理解和执行该原文任务，不能替代、改写、总结或转换用户原文。
- 长上下文结束后必须再次用简短 `task_reminder` 收束回 `user_request`，避免模型在大量前端可见配置、历史消息、附件、检索源说明中丢失当前任务方向。
- 逐项说明传给 Claude SDK 的上下文字段来源：历史消息、引用消息、图片描述、附件、知识库、数据源、Web、MCP、Skills、产物配置等。
